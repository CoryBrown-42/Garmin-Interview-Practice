Mastering Tile-Based Rendering (TBR)

TBR Interview Glossary

Binning Pass (Geometry)

The GPU collects all geometry for the frame and sorts it into screen-space tiles (e.g. 32x32px).

On-Chip Memory (SRAM)

TBR's superpower. Color, depth, and stencil are stored in fast local memory, avoiding expensive battery-draining DRAM access.

Hidden Surface Removal (HSR)

Unlike desktop GPUs that Z-test during shading, TBR GPUs (like PowerVR) can discard occluded pixels before fragment shading begins.

Tile Flush / Memory Bandwidth

The goal is to NEVER flush a tile back to memory mid-frame. This is why "Render-to-Texture" is expensive on mobile.

Mobile Shader Optimization

  • 1

    Avoid Alpha Test: On many TBR GPUs, `discard` instructions kill the ability to use HSR/Z-Prepass optimizations. Use Alpha Blending where possible, or keep Alpha Test minimal.

  • 2

    Lower Precision: Use `half` for colors and normals; only use `float` for texture coordinates or high-precision math to save register space.

  • 3

    Minimize Branching: Dynamic branches kill GPU parallelism. Use `lerp()` or `step()` instead of `if` statements where possible.

  • 4

    Batch State Changes: Sort draw calls by material/shader. Every state change stalls the pipeline on embedded GPUs.

Advanced Shader Techniques for Embedded

üî• Performance-Critical Patterns

Texture Atlasing Strategy

On embedded GPUs, texture switches are catastrophic. Combine all UI elements into a single 2048x2048 atlas. Use a channel-packed approach: RGB for albedo, A for signed distance fields.

Signed Distance Field (SDF) Rendering

Perfect for aviation symbology. SDFs scale infinitely without aliasing and require only a single texture sample. Ideal for altitude tapes, heading indicators, and flight path markers.

Constant Buffer Packing

Align your cbuffer to 16-byte boundaries. Pack `float3 + float` together. Never waste a register‚Äîon mobile, you have precious few.

‚ùÑÔ∏è Memory Bandwidth Savers

Framebuffer Fetch (FBF)

On TBR GPUs, you can read the current pixel value for free using `gl_LastFragData` (GLES) or `[[color(0)]]` (Metal). Use this for blending operations instead of separate render passes.

Subpass Dependencies (Vulkan)

Group dependent render operations into a single render pass with subpasses. The driver can keep intermediate results in tile memory, avoiding costly DRAM writes.

MSAA Resolve Strategy

On TBR, MSAA is nearly free if you resolve at the end of the render pass (on-chip). Never resolve mid-frame. For aviation displays, 4x MSAA is often sufficient.

Graphics & Rendering

"Explain the difference between forward and deferred rendering. Which would you use for an aviation HMI?"

‚úÖ Answer:

Forward rendering processes each object through the full pipeline (vertex ‚Üí fragment ‚Üí output) immediately. Deferred rendering first renders geometry data to G-buffers (position, normal, albedo), then applies lighting in a separate pass.

For aviation HMI: Forward rendering. Here's why:

  • ‚Ä¢ Lower memory bandwidth ‚Äî No G-buffer writes/reads, critical on embedded TBR GPUs
  • ‚Ä¢ MSAA-friendly ‚Äî Deferred struggles with MSAA; forward gets it nearly free on TBR
  • ‚Ä¢ Transparency support ‚Äî HMI has lots of alpha-blended elements; deferred requires separate forward pass anyway
  • ‚Ä¢ Simpler debugging ‚Äî Aviation software needs to be auditable and certifiable

"I'd use forward rendering for an aviation HMI. The display is mostly UI with limited dynamic lights, so deferred's benefits don't apply. Forward is simpler, uses less bandwidth, and MSAA works natively on TBR GPUs."

"What causes aliasing in SDFs and how do you fix it?"

‚úÖ Answer:

SDF aliasing occurs when the distance field gradient changes faster than pixel sampling rate. At sharp edges or small features, the SDF value jumps abruptly between pixels, causing jagged edges.

Fixes:

  • ‚Ä¢ Smoothstep anti-aliasing ‚Äî Use smoothstep() based on screen-space derivatives: smoothstep(-fw, fw, dist) where fw = fwidth(dist)
  • ‚Ä¢ Multi-channel SDFs (MSDF) ‚Äî Store separate distance fields for edges in RGB channels; reconstruct sharp corners
  • ‚Ä¢ Supersampling ‚Äî Sample multiple points within the pixel and average (expensive but high quality)
  • ‚Ä¢ Proper mip-mapping ‚Äî Generate SDF mips correctly (not just box filter)

"I use screen-space derivatives with smoothstep for real-time AA. The key insight is that fwidth() tells you how fast the SDF is changing‚Äîyou soften the edge proportionally. For text and icons, MSDF gives you sharp corners at any scale."

"How would you implement efficient anti-aliasing on a TBR GPU?"

‚úÖ Answer:

On TBR GPUs, MSAA is nearly free if done correctly. The key is keeping samples in on-chip tile memory and resolving at the end of the render pass.

Best approach:

  • ‚Ä¢ Use MSAA 4x ‚Äî Sweet spot for quality vs. cost; samples stay in SRAM
  • ‚Ä¢ Resolve at pass end ‚Äî Never resolve mid-frame or you pay full bandwidth cost
  • ‚Ä¢ Avoid MSAA + transparency ‚Äî Alpha-tested geometry kills HSR; use alpha-to-coverage or separate pass
  • ‚Ä¢ Don't use post-process AA (FXAA/TAA) ‚Äî These require reading back the framebuffer, expensive on TBR

Why MSAA is cheap on TBR: Samples never leave the tile. On desktop (IMR), MSAA multiplies framebuffer bandwidth. On TBR, the tile holds all samples in fast SRAM until final resolve.

"On TBR, I always reach for MSAA first. 4x samples fit in tile memory, and resolve happens once at pass end. It's essentially free compared to post-process AA which would require reading back the whole framebuffer."

"What's the cost of a texture sample vs. an ALU operation on mobile?"

‚úÖ Answer:

On mobile GPUs, texture samples are roughly 4-10x more expensive than ALU operations in terms of both latency and power consumption.

Typical costs (relative):

  • ‚Ä¢ ALU (add, mul, mad) ‚Äî 1 cycle, negligible power
  • ‚Ä¢ Texture sample (cached) ‚Äî 4-8 cycles, moderate power
  • ‚Ä¢ Texture sample (cache miss) ‚Äî 100+ cycles, high power + bandwidth
  • ‚Ä¢ Dependent texture read ‚Äî 2x texture cost (can't prefetch)

Implications:

  • ‚Ä¢ Compute procedurally when possible instead of LUT textures
  • ‚Ä¢ Pack multiple values into RGBA channels
  • ‚Ä¢ Avoid dependent texture reads (UV computed from another texture)
  • ‚Ä¢ Use smaller textures + bilinear over larger + point filtering

"I think in terms of texture:ALU ratio. If I can compute something in under 8-10 ALU ops, it's usually cheaper than a texture lookup. That's why I often compute noise or gradients procedurally rather than sampling lookup tables."

"How would you design an HDA that non-technical artists can use safely?"

‚úÖ Answer:

The key is constraining the parameter space while maintaining creative flexibility. Artists should be able to break aesthetics, not the tool.

Design principles:

  • ‚Ä¢ Clamp all parameters ‚Äî Min/max ranges that prevent degenerate geometry (zero-area faces, inverted normals)
  • ‚Ä¢ Sensible defaults ‚Äî Tool should produce valid output with zero customization
  • ‚Ä¢ Progressive disclosure ‚Äî Basic tab with common params, Advanced tab hidden by default
  • ‚Ä¢ Visual feedback ‚Äî Show guide geometry, bounds, warnings in viewport
  • ‚Ä¢ Validation node ‚Äî Internal check that flags issues before export (non-manifold, UV overlap)

UI/UX patterns:

  • ‚Ä¢ Presets dropdown for common configurations
  • ‚Ä¢ Tooltips explaining each parameter's effect
  • ‚Ä¢ "Randomize within safe bounds" button for exploration
  • ‚Ä¢ Clear error messages, not cryptic Houdini cook errors

"I design HDAs like APIs‚Äîthe public interface is clean and safe, the implementation complexity is hidden. Artists should be able to explore freely without ever producing invalid geometry."

"Explain how PDG handles dependencies between work items."

‚úÖ Answer:

PDG (Procedural Dependency Graph) uses a directed acyclic graph (DAG) where work items flow between TOP nodes. Dependencies are tracked automatically based on data flow.

Key concepts:

  • ‚Ä¢ Work Item ‚Äî A single unit of work (one terrain, one texture, one course)
  • ‚Ä¢ Upstream/Downstream ‚Äî Work items inherit attributes from upstream items
  • ‚Ä¢ Partitioning ‚Äî Many-to-one relationships (merge 18 holes into 1 course)
  • ‚Ä¢ Fan-out ‚Äî One-to-many (one course ‚Üí 5 LOD levels)

Dependency types:

  • ‚Ä¢ File dependencies ‚Äî Work item declares input/output files; PDG tracks staleness
  • ‚Ä¢ Attribute dependencies ‚Äî Work item needs attribute from upstream item
  • ‚Ä¢ Order dependencies ‚Äî Explicit "wait for X before starting Y"

Execution: PDG only re-cooks work items whose inputs changed. If hole #7 is modified, only that hole and its downstream dependents re-process‚Äînot all 18 holes.

"PDG is basically a build system like Make, but for content. It tracks file and attribute dependencies, only re-cooks what changed, and can distribute work items across a render farm. Perfect for processing 43k golf courses."

"How do you validate procedurally generated content at scale?"

‚úÖ Answer:

You need automated validation at multiple levels: geometric, visual, and functional. Manual QA doesn't scale to thousands of variations.

Validation layers:

  • ‚Ä¢ Geometric validation ‚Äî Check for non-manifold edges, degenerate faces, UV overlap, vertex count within budget
  • ‚Ä¢ Bounds checking ‚Äî Asset stays within expected bounding box, no floating geometry
  • ‚Ä¢ Visual regression ‚Äî Render from fixed cameras, diff against golden masters, flag if delta > threshold
  • ‚Ä¢ Functional testing ‚Äî For games: can the ball roll correctly? Are physics colliders valid?

Implementation:

  • ‚Ä¢ Validation as a PDG node in the pipeline‚Äîfailures block export
  • ‚Ä¢ Generate HTML reports with thumbnails of flagged assets
  • ‚Ä¢ Statistical monitoring: if >5% of today's batch fails, alert the team
  • ‚Ä¢ A/B comparison: diff new pipeline version against previous

"I built visual regression testing for avatar clothing. We rendered each variation, diffed against golden masters, and flagged deviations. Same approach works for golf courses‚Äîrender each hole from a few angles, diff automatically, human reviews only the outliers."

"What's your approach to versioning procedural assets?"

‚úÖ Answer:

Procedural assets have two versioning concerns: the tool version (HDA, scripts) and the output version (generated assets). Both need tracking.

Tool versioning:

  • ‚Ä¢ Semantic versioning for HDAs ‚Äî Major.Minor.Patch with clear compatibility rules
  • ‚Ä¢ Version embedding ‚Äî Store tool version in output asset metadata
  • ‚Ä¢ Backward compatibility ‚Äî Old scenes should cook with new tools (or gracefully fail)
  • ‚Ä¢ Version locking ‚Äî Production can pin to a specific tool version

Output versioning:

  • ‚Ä¢ Content-addressable storage ‚Äî Hash the inputs; same inputs = same output
  • ‚Ä¢ Deterministic generation ‚Äî Same seed + params must produce identical output
  • ‚Ä¢ Manifest files ‚Äî Track which tool version + input hash produced each asset
  • ‚Ä¢ Regeneration strategy ‚Äî Can you rebuild old assets from source? How far back?

"I treat procedural tools like code‚Äîsemantic versioning, changelog, deprecation warnings. For outputs, I use content hashing so we can detect when something actually changed vs. just rebuilt. It's critical for incremental builds and debugging regressions."

Architecture & Process

"How do you balance technical debt against feature velocity?"

‚úÖ Answer:

Technical debt is a tradeoff, not a failure. The key is making it visible and intentional, not accidental.

My framework:

  • ‚Ä¢ Categorize debt ‚Äî "Reckless" (will bite us) vs. "Prudent" (acceptable shortcut with known cost)
  • ‚Ä¢ Document decisions ‚Äî ADRs (Architecture Decision Records) explaining why we took the shortcut
  • ‚Ä¢ Budget debt work ‚Äî 20% of each sprint for maintenance/refactoring, non-negotiable
  • ‚Ä¢ Measure pain ‚Äî Track time lost to workarounds; when it exceeds fix cost, prioritize

Red lines I don't cross:

  • ‚Ä¢ No debt in safety-critical paths (aviation display rendering)
  • ‚Ä¢ No debt that blocks other teams
  • ‚Ä¢ No debt that makes the problem harder to fix later

"I'm pragmatic about debt‚Äîsometimes shipping faster is the right call. But I document it, budget time to pay it down, and have red lines I won't cross. The worst debt is the kind you didn't know you were taking on."

"Describe your ideal CI/CD pipeline for art assets."

‚úÖ Answer:

Art asset CI/CD needs to handle large files, long build times, and subjective quality‚Äîdifferent challenges than code CI.

Pipeline stages:

  • ‚Ä¢ Commit hooks ‚Äî Validate file formats, naming conventions, metadata before accepting
  • ‚Ä¢ Fast validation (minutes) ‚Äî Geometric checks, reference validation, thumbnail generation
  • ‚Ä¢ Full build (hours) ‚Äî LOD generation, texture compression, all platform variants
  • ‚Ä¢ Visual QA ‚Äî Automated renders, diffed against previous version, flagged for review
  • ‚Ä¢ Integration test ‚Äî Asset loads in engine, no errors, performance within budget

Key features:

  • ‚Ä¢ Incremental builds ‚Äî Only rebuild changed assets + dependents
  • ‚Ä¢ Distributed processing ‚Äî Fan out to render farm for parallel processing
  • ‚Ä¢ Artifact caching ‚Äî Don't rebuild if inputs haven't changed (content hashing)
  • ‚Ä¢ Rollback capability ‚Äî One-click revert to last known-good asset version

"Art CI is about fast feedback loops. Artists should know within minutes if their asset is valid, but full builds can run overnight. The key is incremental processing‚Äî43k golf courses can't rebuild from scratch on every commit."
